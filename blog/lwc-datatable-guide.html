LWC Datatable: Building a Reusable Paginated Component

By Sai Sagar Chilakapati · December 2025 · 6 min read

Salesforce's out-of-the-box lightning-datatable is powerful, but once you're dealing with hundreds or thousands of rows, you need pagination, search, and row selection to keep the user experience smooth. That's exactly why I built LWC Datatable — a generic, plug-and-play component that wraps these features into a reusable package.

The Problem

The standard lightning-datatable renders all rows at once. For datasets beyond 100 rows, this leads to slow rendering, unresponsive scrolling, and a poor experience on mobile devices. You could implement infinite scroll, but for many enterprise use cases, explicit pagination gives users more control and predictability.

The Solution: lwcDatatableUtility

The component is designed as a generic utility — you pass it data, columns, and configuration, and it handles pagination, row selection with static headers, and a built-in global search. Here's how to use it:

Step 1: Deploy the Component

Clone the repository and deploy lwcDatatableUtility to your Salesforce org. This gives you the reusable base component.

Step 2: Plug It Into Your Component

In your parent component, declare your data and column definitions, then include the utility in your template: 

<c-lwc-datatable-utility
  records={allRecords}
  total-records={allRecords.length}
  columns={columns}
  key-field="Id"
  show-search-box="true"
  max-row-selection={allRecords.length}
  onpaginatorchange={handlePaginatorChange}
  onsetselectedrecords={handleAllSelectedRows}>
</c-lwc-datatable-utility>

Key Features

  • Pagination — Automatically splits large datasets into pages with configurable page sizes.
  • Row Selection — Multi-select with checkbox support, persisted across page changes.
  • Static Headers — Column headers remain visible as users scroll through data.
  • Global Search — Filter across all columns instantly from a single search input.

Architecture Decisions

I chose to keep the component truly generic — it makes no assumptions about your object model or data source. All configuration is passed in via public properties, and events bubble back up for the parent to handle. This means you can reuse it across Case management, Account lists, custom objects, or any data set.

Build components that are configurable, not opinionated. The more assumptions you bake in, the fewer places it can be reused.
